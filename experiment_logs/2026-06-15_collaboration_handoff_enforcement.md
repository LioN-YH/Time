# 协作规范：上下文 handoff 强制流程

日志日期：2026-06-15 02:14:44 CST

## 目的

强化当前工作区在长任务、长窗口和高上下文占用场景下的 handoff 约束，避免 Codex 在上下文接近 65% 后仍继续埋头推进，导致后续接手缺少清晰状态。

## 背景

用户指出此前 `AGENTS.md` 中已有“上下文消耗接近 65% 左右时应准备 handoff”的描述，但实际 goal 窗口执行任务时没有严格触发，而是继续推进主要任务。需要把该规则从偏好性表达改成更明确的强制流程，并补充可直接填写的 `HANDOFF.md` 模板。

## 操作

1. 阅读并确认当前根目录 `AGENTS.md` 中已有“上下文过长时主动切窗”规则，但原表述仍偏向“就应开始准备”，缺少必须暂停主要任务、必须写入 handoff 文件和 handoff 后切换窗口的硬约束。
2. 修改 `AGENTS.md` 的“协作执行偏好补充”部分，将该条扩展为三条强制要求：
   - 接近 65% 或推理质量明显下降时，必须把 handoff 作为最高优先级步骤；
   - 写完 `HANDOFF.md` 后，应说明触发条件、文件路径和建议继承方式，再切换新窗口或新线程继续；
   - 无法精确读取上下文占用时按保守估计提前触发。
3. 新增根目录 `HANDOFF.md` 模板，包含触发信息、当前目标、已完成步骤、正在运行或已停止的命令、关键路径、失败点和风险、下一步命令、验证口径、日志和文档同步等栏目。
4. 更新 `WORKSPACE_STRUCTURE.md`，把 `HANDOFF.md` 纳入根目录长期文件，并更新文档日期。
5. 按实验日志规范新增本日志，并同步更新 `experiment_logs/README.md` 总览表。
6. 2026-06-15 02:26:33 CST 根据用户追加要求，继续在同一篇日志中记录补充修改：在 `AGENTS.md` 新增“长时间任务优先后台 launcher”规则，要求长跑实验和数据处理优先通过 launcher、`nohup`、`setsid`、`tmux` 或等价后台方式启动，并记录输出目录、主日志、状态文件、PID、监控命令、停止命令和接手方式。

## 结果

`AGENTS.md` 已将 65% handoff 规则改为强制流程；根目录已新增 `HANDOFF.md` 模板；`WORKSPACE_STRUCTURE.md` 已登记该长期协作文档。后续又补充了长时间任务优先后台 launcher 的执行规范，减少长跑任务对当前交互式窗口的依赖。该修改不能让 Codex 产品本身在没有外部自动化的情况下自动开新窗口，但能显著提高后续 agent 在本工作区内识别和执行 handoff、后台运行与接手记录的优先级。

## 结论

本次修改把“接近 65% 时准备 handoff”的软约束升级为“必须暂停主要任务并整理 handoff”的硬性工作区规范。后续长任务如果无法读取精确上下文比例，也应按保守估计提前 handoff，不能等到自动压缩或上下文接近耗尽后再补救。对于预计长时间运行的实验或数据处理，应优先使用后台 launcher 并保留可恢复状态，避免当前窗口中断影响任务本身。

## 下一步方案

后续如需进一步自动化，可考虑在 Codex 外层工作流或 App Server 层监听 token usage 更新，到阈值后触发写入 `HANDOFF.md`、提示 `/fork` 或新线程继承；本仓库当前先保留文档级强约束和模板，不新增自动化脚本。
